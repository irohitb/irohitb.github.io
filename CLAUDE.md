# Portfolio — Coding Rules

Static portfolio site. **Astro** for SSG + a single **React** island (theme
toggle). These rules are binding for all work in `portfolio/`.

## 1. Data is NEVER hardcoded in components (most important)

- **All content and data lives in `src/data/*.yml`**, one file per domain (`profile.yml`, `work.yml`, `music.yml`, `health.yml`, `github.yml`, …). Group by what the data *is*, not which component uses it. Components contain *zero* literal content — no names, copy, numbers, URLs, list items inline in markup.
- Each YAML is read by a sibling **typed accessor** `src/data/<domain>.ts` that exports typed values via the shared `loadYaml(file, fallback)` helper (`src/data/_load.ts`). Define an explicit `interface` per dataset so the shape is the contract. Components **import** from the accessor and render it — a component reads like a template; swapping data never touches a component.
- YAML files are either **hand-edited** (work, music, profile, life, recommendations, offTheClock) or **generated by a script** (`health.yml` ← `build-health.mjs`, `github.yml` ← `build-github.mjs`). The update model is always "update the YAML" — either edit it or let a script/CI rewrite it. Generated files carry a "do not edit by hand" header.
- Computed values (e.g. birthday age/countdown in `life.ts`) live in the accessor and read their inputs (`dateOfBirth`) from YAML.

## 2. Every external source sits behind a typed accessor (the API seam)

- Anything sourced from an API is fetched by a **build script** that writes the domain's YAML (e.g. `build-github.mjs` → `github.yml`); the accessor (e.g. `getContributions(): number`) only ever reads that YAML. The API touches the YAML, never the component.
- The accessor is the **only** place that reads a given file, and every read goes through `loadYaml` with a fallback. A missing/malformed YAML or a failed fetch falls back to last-known data — it must never break the build.
- Generator scripts are best-effort: on missing token or fetch error they leave the existing YAML untouched, so the site keeps its last-known values.
- Document each source in a header comment: current status (hardcoded vs live), the future API, and the credential needed.

## 3. No secrets in code

- API keys/tokens come from `process.env.*` only, read inside the accessor. Never commit secrets. `.env` is gitignored; document required vars in the README.

## 4. Styling: tokens only, no raw theme values

- Colors/theme values come from the CSS variables in `src/styles/global.css` (`var(--token)`). Never hardcode a theme hex in a component.
- Theme switching is via the `data-theme` attribute on `<html>`. Adding a themeable value means adding a token in both `[data-theme="dark"]` and `[data-theme="light"]`.

## 5. Minimize client JS

- Default to static Astro components (zero JS). Only ship a React island when genuine interactivity is required, and keep islands small. Today the only island is `ThemeToggle`.

## 6. TypeScript

- `strict` mode. **No `any`.** Type the data shapes. Build must pass `astro check`.

## 7. Freshness model

- Static site → data is fetched at **build time**; there is no per-request server. "Live" means rebuilding (manually or via scheduled CI). Don't introduce a runtime server unless a feature truly needs per-visit freshness.

## Structure

```
portfolio/
  src/
    data/         # ALL content + API accessors (rule #1, #2)
    components/   # presentational .astro + minimal React islands
    layouts/      # Base.astro (head, fonts, theme bootstrap)
    styles/       # global.css (theme tokens)
    pages/        # index.astro (assembles sections)
  public/assets/  # images (logos, avatars)
```
